# Sales Fact Improvement Plan

A phased roadmap for hardening and upgrading `src/facts/sales`. Derived from a deep
architectural review (43 verified findings across 12 concern areas, after an
adversarial pass rejected 31 weaker/incorrect ones).

**Guiding principle:** the code is competent and mostly correct. These phases turn
"good enough" into "right by construction" ‚Äî fixing stated guarantees that don't
hold, then raising statistical realism, then paying down structural debt. Order is
chosen so each phase is *safe to land on its own* and the riskiest behavioral
changes are protected by tests written first.

---

## How to use this document

- Work top-to-bottom. Phases are ordered by **dependency and risk**, not by size.
- Each task is a checkbox. Tick it only when its **acceptance criteria** pass.
- `Finding #N` cross-references the review. Severity in the review was: 5 significant,
  16 moderate, 20 minor, 2 cosmetic (after adversarial downgrade).
- Don't start Phase N+1 until Phase N's **exit criteria** are green ‚Äî especially
  Phase 0, which is the safety net for everything after it.

### Status legend
`[ ]` not started `[~]` in progress `[x]` done `[!]` blocked `[-]` dropped/out of scope

### Effort / risk tags
`E:S|M|L` effort (small <0.5d, medium ~1‚Äì3d, large >3d) ¬∑ `R:lo|med|hi` behavioral risk

## Phase 0 ‚Äî Guardrails (write the tests first)

**Why first:** Phases 1‚Äì4 change generated bytes. Without regression guards we can't
tell an intended change from an accidental one. These tests also *encode* the
guarantees the project advertises, so they double as living documentation.

- [x] **0.1 Cross-worker-count determinism test** ‚Äî `E:M R:lo` ‚Äî **DONE**
  `tests/test_sales_determinism_workers.py`: generates a small sales fact at
  `--workers 1` vs `--workers 4` (same seed/config), packages each, sorts by
  `(OrderNumber, OrderLineNumber)`, and compares a SHA-256 of the rows. Three tests:
  fixed-worker-count determinism (passes), single-chunk worker-invariance control
  (passes), multi-chunk worker-invariance (**xfail strict** ‚Äî Finding #5/#6).
  - *Run:* `uv run --extra dev pytest tests/test_sales_determinism_workers.py`
    (pytest lives in the `dev` extra; the `.venv` is uv-managed, no pip).
  - *Empirical notes for Phase 1.1 (verified while authoring):*
    - Discovery **cannot be disabled via config** ‚Äî the auto-adjust at
      `sales.py:~2436` raises `new_customer_share` above 0 whenever undiscovered
      customers exist (even the `stagnation` preset gets bumped to ~0.36‚Äì0.50). So
      the *single-chunk* run is used as the "discovery-on but still invariant"
      control instead of a discovery-off run.
    - Divergence is **chunk-count sensitive**: at ~6 chunks discovery saturates
      within each worker's slice and the arrangements reconverge (no divergence);
      at **‚â•8 chunks it reliably diverges**. The test uses 12 chunks
      (12k rows / 1k chunk / 4k customers) ‚Üí diverged on 3/3 runs (~93% of rows),
      single-chunk control identical on all.
    - **Acceptance flip:** when Phase 1.1 lands, the multi-chunk test XPASSes; the
      `strict=True` marker turns that into a failure ‚Üí delete the `xfail` marker to
      make it a permanent hard regression guard.
- [x] **0.2 Chunk-size invariance test** ‚Äî `E:M R:lo` ‚Äî **DONE**
  `tests/test_sales_determinism_chunksize.py`: fixes `--workers 1` (so `chunk_size`
  is the only variable ‚Äî no worker-scheduling noise) and compares a single-chunk run
  vs a 12-chunk run of the same config + seed. Two tests: total-row invariance
  (passes), per-month distinct-customer invariance (**xfail strict** ‚Äî Finding #4/#14).
  - *Empirical notes for Phase 2 (verified while authoring):* total rows **and**
    total distinct customers were unchanged across chunk sizes, but the **per-month
    distinct-customer curve shifted by up to ~48 customers/month** at 12 chunks ‚Äî i.e.
    chunking redistributes *which* customers transact *when*, so `base_distinct_ratio`
    is chunk-size-coupled. Because both runs are single-worker the divergence is fully
    deterministic (no flakiness), so the strict xfail is rock-solid.
  - *Acceptance flip ‚Äî DONE:* Phase 2 landed; the per-month test XPASSed and the
    `xfail` marker was removed, so it is now a hard regression guard (asserting the
    per-month distinct-customer curve is byte-identical across `chunk_size`).
  - *Shared harness:* the small end-to-end generation helpers now live in
    `tests/sales_gen.py` (config builder, sales-fact loader, digest, per-month
    distinct), reused by 0.1, 0.2, and future phase-acceptance tests.
- [x] **0.3 Schema-consistency test** ‚Äî `E:S R:lo` ‚Äî **DONE**
  `tests/test_sales_schema_consistency.py` (fast pure-unit): compares the Arrow output
  schema (`sales_worker/schemas.py`) against the SQL schema (`static_schemas.py`) for
  both `skip_order_cols` settings. Names + order + dtype-*family* agree (**passes**);
  exact dtypes **xfail** ‚Äî `ChannelKey`/`TimeKey` are `SMALLINT` (int16) in SQL but
  `int32` in parquet, the real Finding #15 drift.
  - *Note:* the worker-config TypedDict is the worker *input* contract, not the Sales
    *output* column set, so it is (correctly) not compared here.
  - *Acceptance flip:* when Phase 5.5 unifies to one canonical `LogicalType`, the
    exact-dtype test XPASSes ‚Üí delete the `xfail` marker.
- [x] **0.4 Golden-fixture regeneration doc** ‚Äî `E:S R:lo` ‚Äî **DONE**
  `tests/README.md`: documents running the suite (`uv run --extra dev pytest`), the
  three Phase-0 guardrails and which phase flips each xfail, the **xfail ‚Üí hard-assert
  protocol**, and a regeneration recipe using `tests/sales_gen.py`.
  - *Reality check:* there are **no stored golden fixtures** today ‚Äî the guardrails
    compare two in-process runs, so nothing on disk needs regenerating. The doc records
    that and gives the `sales_gen` recipe for when a committed reference digest is ever
    needed (e.g. after Phases 3‚Äì4 change outputs).

**Exit criteria:** ‚úÖ 0.1‚Äì0.3 exist and run green (passes + tagged strict-xfails); the
xfail‚Üíhard-assert protocol is documented (0.4) so later phases flip them deliberately.
**Phase 0 complete.**

## Phase 1 ‚Äî Determinism & correctness

**Goal:** every stated guarantee ("deterministic, idempotent, independent of worker
count") becomes true. These are the *foundational* fixes.

- [x] **1.1 Closed-form customer discovery** ‚Äî `Finding #5 #6 #18` ¬∑ `E:L R:hi` ‚Äî **DONE**
  Replaced the mutable per-worker `State.seen_customers` accumulator with a deterministic
  `customer_discovery_month[i] = clip(start_month_i + hash_lag(CustomerKey_i, run_seed), start_i, end_i)`
  computed **once** in the coordinator and broadcast read-only into shared memory like the
  other customer arrays. In month `m` the chunk builder now force-introduces the cohort
  scheduled exactly at `m` (`discovery_month == m`), treats `discovery_month <= m` as the
  repeat pool, and never samples not-yet-introduced customers. Discovery is now a pure
  function of each chunk's own inputs, so worker count no longer affects output.
  - *Files:* `sales_logic/core/customer_sampling.py` (`compute_discovery_months`,
    `_hash_uniform`, rewritten `_sample_customers` discovery branch),
    `sales_logic/chunk_builder.py` (read schedule, drop per-chunk target/shape math),
    `sales.py` (coordinator pre-pass + publish), `sales_worker/init.py` (bind),
    `sales_logic/globals.py` (drop `seen_customers`, add `customer_discovery_month`),
    `worker_cfg_schema.py`, `config_schema.py` (new `discovery_lag_scale` knob, default 1.0).
  - *Deleted:* `_make_seen_lookup`, `_update_seen_lookup`, `_build_seen_mask`,
    `_SPARSE_KEY_RATIO`, the `State.seen_customers` carve-out, the old `discovery_cfg`/
    `_target_new_customers`/`stochastic_discovery`/bootstrap/`discovery_shape` per-chunk path.
  - *Acceptance ‚Äî met:* test **0.1 flipped to a hard pass** (marker removed; now asserts
    workers 1/2/4 agree). Empirically verified worker-invariance across 3 seeds √ó workers
    {1,3,5} √ó `skip_order_cols` {False,True} (18 configs, all byte-identical). The emergent
    acquisition curve tracks the shipped one within tolerance (m=0 clip-spike smoothed
    308‚Üí187 exactly as before; total coverage identical at 3800/4000; mid-peak ~208) with a
    mild, benign right-censoring bump in the final month.
  - *Companion fix (uncovered by the stress once discovery stopped masking it):* the
    worker-lifetime product CDF cache in `_sample_brand_aware` was keyed by **calendar**
    month while its brand-mix value depended on the **absolute** month; when month-skipping
    chunks landed on different workers the first chunk to touch a calendar month fixed the
    wrong brand mix for later same-calendar-month months (~0.2% of rows on some seeds).
    Re-keyed the cache by `m_offset`. Guarded by `test_multi_chunk_worker_invariance_second_seed`
    (seed 20250701, which triggers it; seed 1234 does not).
- [x] **1.2 Single OrderNumber int64 decision** ‚Äî `Finding #21` ¬∑ `E:M R:med` ‚Äî **DONE**
  The int64-vs-int32 choice is computed **once** from the real emitted ID ceiling
  (`_order_id_int64` in `sales.py`), stored on `SalesRunManifest.order_id_int64`, and
  threaded (sales_runner ‚Üí `package_output` ‚Üí `write_create_table_scripts` ‚Üí
  `generate_all_create_tables`) so the SQL DDL applies it via a new `force_int64` arg to
  `get_sales_schema`/`promote_order_number` across **all** order-number-carrying tables
  (Sales/Header/Detail/Returns/Complaints). The `total_rows` fallback (no manifest) is now
  **id-space aware** ‚Äî `order_id_int64_for_rows()` promotes at ~134M rows (the ~8√ó day-band
  factor), matching the parquet writer, instead of the old ~1.07B raw threshold.
  - *Files:* `static_schemas.py` (`order_id_int64_for_rows`, `force_int64`), `sales.py`
    (manifest field + thread), `sales_runner.py`, `packaging/package_output.py`,
    `packaging/sql_scripts.py`, `sql/generate_create_table_scripts.py`.
  - *Acceptance ‚Äî met:* a 200M-row config (the old 134M‚Äì1.07B mismatch band) now emits
    BIGINT in the DDL, matching parquet. `TestOrderIdInt64PathsAgree` asserts the Arrow and
    SQL widths agree across a total_rows range; `TestOrderNumberDDLPromotion` adds the
    mismatch-zone regression + authoritative-flag-override guards. `promote_order_number`'s
    raw-threshold semantics (and its unit tests) are preserved for flagless callers.
- [x] **1.3 Single inflation epoch** ‚Äî `Finding #30` ¬∑ `E:S R:lo` ‚Äî **DONE**
  `_global_start_month_int()` resolves the inflation anchor once from `State.date_pool` (the
  configured dataset start), memoized per worker; the per-chunk `min(order_dates)` fallback is
  deleted (missing date_pool now raises `SalesError` rather than silently anchoring per-chunk).
  - *Files:* `sales_models/pricing_pipeline.py`.
  - *Acceptance ‚Äî met:* no `min(order_dates)` fallback remains; `TestGlobalStartMonth` asserts
    the anchor is date_pool-derived and identical across chunks (independent of a chunk's own
    order dates).
- [x] **1.4 Unify per-chunk seeding** ‚Äî `Finding #33` ¬∑ `E:S R:lo` ‚Äî **DONE**
  `derive_chunk_seed(run_seed, idx)` now uses the house pattern
  `SeedSequence(run_seed).spawn(idx+1)[idx].generate_state()` (identical child to
  `spawn(n_chunks)[idx]`). The coordinator's materialized `rng_master`/`seeds` array and the
  redundant `+ idx*stride` are gone; the task tuple carries the run seed itself.
  - *Files:* `sales.py` (drop `seeds` array), `sales_worker/task.py`.
  - *Acceptance ‚Äî met:* chunk seed is a pure function of `(run_seed, chunk_idx)`, regenerable
    in isolation (`TestDeriveChunkSeed`); output stays deterministic and worker-count
    invariant (re-verified across the 3-seed √ó workers{1,3,5} √ó skip{F,T} stress).

**Phase 1 complete.** Every stated determinism guarantee ("deterministic, idempotent,
independent of worker count") now holds by construction, verified end-to-end.

**Exit criteria:** Phase 0 determinism + chunk tests pass with discovery **on**; no fix
in this phase depends on convention-only invariants.

## Phase 2 ‚Äî Decouple business knobs from chunking

**Goal:** "plan globally, shard the index space, materialize locally." Timeline-shaped
quantities stop being implicit functions of `chunk_size` and worker count.

- [x] **2.1 Global per-month plan** ‚Äî `Finding #14` ¬∑ `E:L R:hi` ‚Äî **DONE**
  The coordinator computes three length-`T` arrays ONCE from the seed against the
  GLOBAL month totals and broadcasts them read-only: `sales_rows_per_month` (R[m], via
  `build_rows_per_month` at the global scale), `sales_orders_per_month` (O[m] = R[m] /
  avg-lines-per-order), and `sales_distinct_target` (D[m]). Each chunk slices a
  contiguous band of every month's order-id space with `_chunk_month_band`: orders
  floor-tile `[0, O[m])` with **no gaps** (`sum == O[m]`) and lines start at one-per-order
  plus a proportional share of the extra, so `n_lines >= n_orders` and per-chunk line
  sums are exact (`total_rows` preserved).
  - *Files:* `sales.py` (global plan pre-pass + broadcast + `total_chunks`),
    `sales_logic/chunk_builder.py` (`_chunk_month_band`, rewritten month loop),
    `globals.py` / `sales_worker/init.py` / `worker_cfg_schema.py` (plan fields).
  - *Acceptance ‚Äî met:* test **0.2 flipped to a hard pass** (per-month distinct AND
    per-month row curve byte-identical across `chunk_size`).
- [x] **2.2 Chunk-invariant distinct-customer target** ‚Äî `Finding #4` ¬∑ `E:M R:med` ‚Äî **DONE**
  `compute_month_distinct_targets` evaluates D[m] against the GLOBAL month totals
  (eligible count + O[m]), not per-chunk rows. A month's distinct-customer pool is built
  once per worker (`build_month_customer_pool`, cached by month offset) and a given
  global order index always maps to the same customer (`assign_orders_to_customers`:
  `pool[j]` for `j < len(pool)`, else a hash-weighted repeat from the pool). So the
  per-month distinct-customer set is **exactly the pool** regardless of chunking ‚Äî repeats
  never introduce anyone new and every distinct-prefix index is materialized by exactly
  one chunk (the no-gap band property).
  - *Files:* `sales_logic/core/customer_sampling.py` (new plan functions),
    `sales_logic/chunk_builder.py` (`_get_month_pool`, month loop).
  - *Acceptance ‚Äî met:* `distinct_ratio` produces the same distinct curve for any
    `chunk_size`. Stress: per-month rows + distinct byte-identical across `chunk_size`
    {6000, 4000, 3000, 1500, 1000, 777, 500, 333}; worker byte-identity at
    `chunk_size=1000` for workers {1, 2, 4}.
- [x] **2.3 Collapse duplicate participation-target logic** ‚Äî `Finding #17` ¬∑ `E:S R:lo` ‚Äî **DONE**
  Deleted the dead `_participation_distinct_target` (and its phantom `cycles` config keys,
  which no config set) ‚Äî `compute_month_distinct_targets` is now the single source of
  truth for the per-month distinct target. The legacy per-chunk sampler `_sample_customers`
  is likewise dead after Phase 2 (live path = `build_month_customer_pool` +
  `assign_orders_to_customers`) and carries a dead-code marker; its removal is folded into
  **Phase 5.1** (State/architecture cleanup) to keep this commit low-risk.
  - *Files:* `sales_logic/core/customer_sampling.py`, `sales_logic/core/__init__.py`.
  - *Acceptance ‚Äî met:* no second, divergent copy of the target math; `cycles` keys gone;
    `TestParticipationDistinctTarget` replaced by `TestComputeMonthDistinctTargets`.

**Exit criteria:** ‚úÖ business knobs (`distinct_ratio`, discovery share) are reproducible and
chunk/worker-invariant; Phase 0 invariance tests are now hard asserts. **Phase 2 complete.**

## Phase 3 ‚Äî Statistical fidelity (the joint model)

**Goal:** the highest-value upgrade for the product's purpose. Replace a bag of
independent marginals with a few latent drivers that make facts *correlate* the way real
retail does. **Each sub-phase intentionally changes output ‚Üí regenerate fixtures (0.4).**

> Implement these as additive, config-gated features (default-on, but toggleable) so the
> change is reviewable and reversible.

**Shared design (all sub-phases):** each new stochastic behavior is either chunk-idx-seeded
off the chunk RNG or **hash-seeded on a stable global key** (ProductID, OrderNumber+line) ‚Äî
never introduced in a way that shifts row counts or customer assignment. So the Phase 0‚Äì2
guardrails (worker byte-identity, per-month rows + distinct-customer chunk invariance) still
hold; only per-line *measures*/identity change. Each feature is a `models.*` toggle
(default-on) validated by unit tests **and** an end-to-end acceptance in
`tests/test_sales_phase3_fidelity.py`.

- [x] **3.1 Quantity conditional on product & price (elasticity)** ‚Äî `Finding #2 #12` ¬∑ `E:M R:med` ‚Äî **DONE**
  `build_quantity` now scales the *float* quantity (before the final round/clamp) by
  `(price/reference)^(-Œµ)` and a per-product unit propensity from `PopularityScore`. The
  reference is the median catalog `ListPrice`, computed lazily from the bound product pool
  (no new broadcast). Pure per-line arithmetic ‚Äî **no RNG**, so determinism is untouched;
  inactive when `unit_price` isn't supplied (unit-test path).
  - *Files:* `sales_models/quantity_model.py` (`_parse_elasticity`, `_reference_price`,
    `_propensity_factor`, restructured `build_quantity`), `sales_logic/chunk_builder.py`
    (thread `product_row_idx`/`unit_price`), `config_schema.py` (`QuantityElasticityConfig`).
  - *Acceptance ‚Äî met:* end-to-end price‚Üîquantity correlation is negative; cheap-half mean
    qty > pricey-half (a TV and a battery pack no longer share a quantity distribution).
- [x] **3.2 Promotion salience weighting** ‚Äî `Finding #3` ¬∑ `E:S R:med` ‚Äî **DONE**
  `apply_promotions` picks among active promotions weighted by
  `salience = exp(Œ≤¬∑DiscountPct)¬∑type_weight` (inverse-CDF sampling) instead of uniformly.
  Per-promo salience is precomputed in the coordinator (`_compute_promo_salience`),
  broadcast as `promo_salience_all`, read in the chunk builder. `None` ‚áí uniform draw.
  - *Files:* `sales_logic/core/promotions.py` (`_weighted_pick`), `sales.py`
    (`_compute_promo_salience` + worker_cfg), `config_schema.py` (`PromoSalienceConfig`).
  - *Acceptance ‚Äî met:* salience on/off comparison ‚Äî the redemption-weighted mean
    `DiscountPct` is higher with salience on (deeper promos redeemed more often).
- [x] **3.3 Basket / co-purchase correlation** ‚Äî `Finding #13` ¬∑ `E:L R:med` ‚Äî **DONE**
  Each order gets a hash-seeded theme (a group of subcategories); a share of its off-theme
  lines are redirected into that theme. Hash-seeded on `OrderNumber`/`OrderLineNumber` (not
  the chunk RNG), vectorized (K-theme loop). `SubcategoryKey` is now loaded whenever present
  and bound read-only on `State`.
  - *Files:* `sales_logic/core/basket.py` (new), `sales_logic/chunk_builder.py` (apply before
    channel enforcement), `sales.py`/`sales_worker/init.py` (subcat publish/bind),
    `config_schema.py` (`BasketConfig`).
  - *Acceptance ‚Äî met:* end-to-end, lines land in their order's theme far above the
    `1/num_themes` chance baseline (recoverable association structure).
- [x] **3.4 Shared fulfillment-friction latent (delivery ‚Üî returns)** ‚Äî `Finding #19 #22` ¬∑ `E:M R:med` ‚Äî **DONE**
  One per-line friction latent ‚Äî a pure SplitMix64 hash of `(OrderNumber, OrderLineNumber)`
  ‚Äî drives delivery bucketing (explicit `(p_early, p_delayed)` + named delay-magnitude
  distribution, replacing the mod-100 ladder) and, recomputed in the separate returns pass,
  return probability (`base¬∑(1+boost¬∑friction)`) and lag (`¬∑(1‚àíshorten¬∑friction)`). Both
  passes read `models.fulfillment` from `State.models_cfg` (no new broadcast).
  - *Files:* `sales_logic/core/delivery.py` (`line_friction`, `_friction_delivery_offset`,
    branched `compute_dates`), `sales_worker/returns_builder.py` (friction fields + coupling),
    `sales_worker/task.py`, `config_schema.py` (`FulfillmentConfig`).
  - *Acceptance ‚Äî met:* delivery status shares track `(p_early, p_delayed)`; a Returns view
    sliced by `DeliveryStatus` shows late ‚Üí more **and** faster returns. Legacy mod-100 path
    + pre-3.4 returns preserved when disabled / at 0 params (returns unit tests unchanged).
- [x] **3.5 Markdown ‚Üî PromotionKey consistency** ‚Äî `Finding #10 (markdown part)` ¬∑ `E:S R:lo` ‚Äî **DONE**
  A discount is now a consequence of a promotion: with reconciliation on
  (`models.pricing.markdown.reconcile_promotions`, default), only promoted lines carry a
  discount (drawn from the *nonzero* ladder). `promo_keys` are threaded into `build_prices`;
  the reconcile draw consumes the same `rng.choice(size=n)` as the legacy lottery.
  - *Files:* `sales_models/pricing_pipeline.py`, `sales_logic/chunk_builder.py`,
    `config_schema.py` (`MarkdownConfig.reconcile_promotions`).
  - *Acceptance ‚Äî met:* `PromotionKey == no_discount_key` ‚áí `DiscountAmount == 0` (strict);
    promoted lines carry a discount (aggregate converse).

**Exit criteria:** ‚úÖ the canonical demo analyses (elasticity, promo lift, market basket,
return-by-delivery) return real, discoverable structure instead of noise. **Phase 3 complete.**

## Phase 4 ‚Äî Pricing model

**Goal:** stop fusing two different concepts into `UnitPrice`. (Dissolves gotcha #26
rather than documenting around it.)

- [x] **4.1 Deterministic posted price** ‚Äî `Finding #10` ¬∑ `E:M R:med` ‚Äî **DONE**
  The posted `UnitPrice`/`UnitCost` snap is now hash-seeded on `(ProductID, month)` instead
  of the per-row chunk RNG, so every sales line for a product-month carries the same posted
  price; per-line variation lives only in `DiscountAmount`/`NetPrice`. Gated by
  `models.pricing.appearance.deterministic` (default on); `product_ids` are threaded into
  `build_prices`. The SCD2 branch (already catalog-price-preserving) is unchanged, so both
  modes now yield a deterministic posted price.
  - *Files:* `sales_models/pricing_pipeline.py` (`_price_hash_u01`, `_pick_ending`,
    `_snap_unit_price`/`_snap_cost` hash params, `build_prices` product_ids),
    `sales_logic/chunk_builder.py`, `config_schema.py` (`AppearanceConfig.deterministic`).
  - *Acceptance ‚Äî met:* end-to-end, `UnitPrice` is single-valued within every
    `(ProductKey, month)`; the OFF path (`deterministic=false` / `product_ids=None`)
    reproduces the legacy per-row stochastic snap. Determinism guardrails still green
    (removing the snap's RNG draws shifts the downstream stream identically across workers).
- [ ] **4.2 (Optional) Category-structured inflation** ‚Äî `Finding #11` ¬∑ `E:L R:med` ‚Äî deferred
  Replace the single global drift with a small factor model (macro + category +
  idiosyncratic). Pair with 3.1's elasticity so markdowns actually cause volume lifts.
  - *Acceptance:* electronics-deflate / groceries-inflate style heterogeneity is visible;
    no determinism/picklability regression.
  - *Note:* lower priority; the elasticity coupling (3.1) already delivers most of the
    value. Deferred as an optional follow-up.

**Exit criteria:** ‚úÖ gotcha #26 rewritten as "posted price is deterministic per
(product, month); discounts vary per line." **Phase 4 core (4.1) complete;** 4.2 optional.

## Phase 5 ‚Äî Architecture cleanup

**Goal:** concerns become objects with contracts. Pure structure/testability ‚Äî **zero
intended behavior change** (use Phase 0 tests as the safety net).

- [x] **5.1 Split State into frozen bundle + scratch; delete sealing theater** ‚Äî `Finding #16 #7` ¬∑ `E:L R:med` ‚Äî **DONE**
  Goal: separate the immutable bound dimension/config data from mutable per-worker
  scratch, kill the dead abstractions, so cross-chunk mutable state can't sneak onto
  `State`. Delivered across five commits (each verified byte-identical via the small-fact
  digest + guardrails):
  - **5.1a** (`7e52c41`) deleted the **sealing theater** ‚Äî `_SealableMeta` metaclass,
    `seal()`, `_sealed`, and `State.validate()` (its one real caller inlined as a

## Out of scope / deliberately NOT doing

From the 31 rejected findings ‚Äî recorded so they don't get re-raised:

- **Don't** assume worker count changes prices ‚Äî it doesn't (only `chunk_size` perturbs
  the stochastic snap). Phase 4.1 makes even that moot.
- **Don't** reuse `_stable_row_hash(date, product)` as a unique line key ‚Äî it collides by
  design (same product+day ‚Üí same value). Any real link key must be an emitted column.
- **Don't** delete the `isinstance(..., Mapping)` branches ‚Äî they're **live** via
  `Mapping.register(_Base)`.
- **Don't** swap Poisson for negative-binomial expecting realism ‚Äî output is clamped to
  [1,8], which censors the tail NB exists to capture. Tune Œª/`max_qty` instead.
- **Don't** add an LP/min-cost-flow solver to coverage repair ‚Äî adds a dependency and a
  determinism risk; a principled deterministic greedy (6.7) gets ~all the value.
- **Don't** treat Delta `partition_by` as a "blind default" ‚Äî it's a documented,
  user-configurable knob (`config.yaml`), with a dedicated compaction tool.
